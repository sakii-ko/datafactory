# Matrix-Game 3.0 数据系统拆解(一手来源:官方技术报告)

> 来源:`SkyworkAI/Matrix-Game` 仓库内 `Matrix-Game-3/assets/pdf/report.pdf`(56MB,20 页),第 4 节 "Data System" 全文抽取。这是公开资料里对其数据制造**披露最细**的部分,本文逐条记录,作为我们数据农场设计的直接依据。

## 0. 总览:三源 + 统一标注/过滤

M-G3 的数据系统 = **三个数据源**喂一个**统一标注与过滤**层:

1. **Unreal Engine 第一人称生成**(他们叫 `Unreal-Gen`)—— 自建合成,真值完美。
2. **可扩展 AAA 游戏录制**,产出**动态第三人称**数据 —— 四层解耦架构。
3. **真实世界数据采集** —— 下载现成数据集 + 统一重标注。
4. **统一标注与过滤** —— caption + 轨迹/速度/质量过滤。

一个关键结论先放这:**「动作」不是从游戏里读出来的,而是从相机/玩家的逐帧位姿推断出来的**(见 4.2 的 WSAD inference)。这把"必须拿到游戏真值动作"这个最大难点绕开了。

## 1. 数据样本的形态(最该抄的 schema)

UE 路线里每个 tick 同步采集一个元组(报告式 (12)):

```
D_t = (I_t, p_t, r_t, c_t, θ_t, a_t)
```

- `I_t ∈ R^{H×W×3}`:渲染出的 RGB 帧
- `(p_t, r_t)`:玩家(角色)的世界**位置 + 旋转**
- `(c_t, θ_t)`:相机的 **6-DoF 位姿**
- `a_t ∈ {0,1}^6`:**6 维离散动作向量**

> **tick-level synchronization**:以上全部在**同一个引擎 tick 回调里**采样 → **时间对齐误差严格为 0**(报告强调这是"外部录制方案无法达到的")。旋转用**双精度四元数**保证亚度级精度。

动作注入模型的方式(供我们对齐 schema):**离散键盘动作走 Cross-Attention,连续鼠标信号走 Self-Attention**(沿用 Matrix-Game 2.0 / GameFactory)。

## 2. 4.1 Unreal-Gen:UE 合成数据生产(第一人称为主,也支持第三人称)

- **场景**:>1,000 个自建 UE5 场景,基于 **Nanite**(虚拟几何)+ **Lumen**(全局光照),电影级画质。
- **自主探索 = NavMesh–RL 混合智能体**(两级):
  - 高层 **RL 策略** `π_θ(g_t | s_t)` 选导航目标,内在奖励 = **覆盖率 bonus + 场景丰富度**;
  - 低层 **NavMesh planner** 算无碰撞路径;
  - 兜底策略:方向性覆盖、shape route 生成、多半径重试;
  - **三重卡死检测**:位置增量、路径超时、包围盒。
  - → 在开放地形 / 密集室内 / 风格化世界都能跑,**无需逐场景调参**。
- **相机 & 角色多样性**:
  - 偏航随机化(360° 全扫 或 8 方向离散)+ 俯仰随机化,与导航并发;
  - **同时支持第一人称 / 第三人称**;
  - **模块化角色装配系统**:把角色拆成可替换部件——上衣 / 裤子 / 鞋 / 头发 / 外套 / 帽子 / 配饰,组合数 `|C| = Π_i |S_i| > 10^8`,运行时采样 → 巨大视觉多样性。
- **工业级自动化(zero-human-in-the-loop)**:
  - 批量渲染:**每张 GPU 多个并发 UE5 实例 + shader warmup**;
  - 实时云同步;
  - 自动 QA:**重复帧检测、位置异常修复、不完整场景检测**;
  - webhook 闭环监控 → **一个操作员看管几十台录制机**。

## 3. 4.2 AAA 游戏录制:四层解耦架构(动态第三人称的来源)

目标:借用商业游戏的视觉丰富度。覆盖游戏:**GTA V、Red Dead Redemption 2、Palworld、Cyberpunk 2077、Hogwarts Legacy**。

**四层**:

1. **Game Process Layer**:每个游戏作为独立进程跑,**注入 agent 插件**做角色控制 + 状态提取。
2. **Agent Layer**:基于 NavMesh 的自主探索 + 多状态机协调 + 8 方向视角切换;通过 **JSON 文件 IPC(<1ms 轮询)**导出逐帧状态
   `s_t = {p_t, r_t, c_t, θ_t, a_nav, a_jump, a_attack}`。
3. **Recording Coordination Layer**:**OBS Studio** 分段录制(60 秒一段)经 **WebSocket** 控制;核心是 **physics-based WSAD 推断**——把位置增量投影到相机局部坐标系:
   ```
   f = (cosθ_yaw, sinθ_yaw),  r = (sinθ_yaw, −cosθ_yaw)
   ```
   再按 `⟨Δp_t, f⟩` 和 `⟨Δp_t, r⟩` 分到 8 个基本方向 → **完全消除人工标注偏差**(动作是算出来的,不是读出来的)。
4. **Dataset Output Layer**:**MP4 视频 + 逐帧 CSV**(6 维动作向量 + 相机参数 + 完整位姿)。

**可扩展性 & 可靠性**:
- **加一个新游戏只需实现该游戏的 Agent Layer**,录制层和输出层完全复用 → 这是整套架构能扩展的关键。
- 全链路可靠性:卡死检测自动恢复、分段录制断线重连、webhook 远程告警。
- 跨 session 随机化环境参数(**天气 / 时间 / NPC 密度**)。
- **总体数据准确率 >99%**。

> ⚠️ 注意:向商业游戏注入插件 / 修改进程 / 录制用于训练,**普遍违反这些游戏的 EULA/ToS**,且每个游戏的注入手段(GTA V 用 ScriptHookV 之类)各不相同、工程量大。报告没谈合规。**这是我们要谨慎评估的一条路**(见 spec 的取舍)。

## 4. 4.3 真实世界数据(纯下载 + 重标注,最省事)

四个现成数据集,互补覆盖室内/户外/街景/航拍车载:
- **DL3DV-10K**:>10,000 条 4K 视频,65 类 POI。
- **RealEstate10K**:室内房产漫游,纯静态场景,相机轨迹极干净。
- **OmniWorld-CityWalk**:YouTube 第一人称城市步行,多天气/光照,相对位姿用 **DPVO** 估计。
- **SpatialVid-HD**(最大子集):行人/驾驶/无人机航拍。

关键做法:虽然有些数据集自带位姿标注,他们**统一用 ViPE 重新标注所有真实视频**,消除跨源坐标约定/位姿表示不一致。

## 5. 4.4 标注与质量过滤

- **文本标注**:每个 clip 用 **InternVL3.5-8B** 生成结构化描述,四层级 schema:
  1. narrative captions(整体语义);
  2. static scene captions(把场景外观与相机运动解耦);
  3. dense temporal captions(逐段事件 + 相机运动);
  4. perceptual quality scores(0–10,五维:运动平滑度、背景动态、场景复杂度、物理合理性、整体质量)。
- **轨迹 & 速度过滤**(三个互补过滤器,全过才保留):
  1. 局部几何一致性(深度重投影误差);
  2. 全局运动异常(最大位移/中位位移比);
  3. 相机速度过滤(中位速度,去掉过慢/过快 clip)。
  - 阈值**在 UNREAL 真值上标定**,再按数据集应用。
- 加上感知质量过滤,整体**剔除 20% 原始数据**。

## 6. 与建模相关、但影响数据设计的点

- **视角专门化**:第一人称和第三人称动态难以联合建模,所以他们**分别训练两个 high-noise 模型**(FPV / TPV),共享一个 low-noise 模型。→ **我们造数据时也应把 FPV / TPV 分开打标签和分桶。**
- **解耦动作与画质**:high-noise 模型用"动作精确"的数据(UE/AAA,带准动作)训练;low-noise 模型可用纯网络视频(无动作)训练以提泛化。→ **数据集要区分"带精确动作"与"仅视频"两档。**
- MG-LightVAE 训练用了 **700K 个 17 帧 clip**(游戏 + 真实混合)。
- 基座:Wan2.2-TI2V-5B,动作模块进前 15 个 DiT block。

## 7. 给我们数据农场的直接启示

1. **样本 schema 直接照抄** `D_t = (RGB, player_pos, player_rot, cam_pos, cam_rot/6DoF, action_6d)`,tick 内同步,MP4 + 逐帧 CSV/JSON。
2. **动作靠位姿推断**(WSAD inference),不依赖游戏真值动作 —— 这是可复制、可在任何"能拿到相机位姿"的来源上用的通用技巧。
3. **三条数据源,工程难度/合规风险从低到高**:真实视频(下载+ViPE 重标注)< UE 自建(Unreal-Gen,真值完美、可控)< AAA 录制(违反 ToS、逐游戏定制)。
4. **农场的"骨架"是引擎无关的**:导航/探索智能体、tick 同步采集、装配多样性、自动 QA、统一标注与过滤、分段输出 —— 渲染器(UE / 真实视频 / 其它)是可替换后端。
5. **FPV/TPV 分离**、**带动作/仅视频分档**,从数据结构层面就要分开维护。
